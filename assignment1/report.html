Normal Use Cases

Member Validation

Use Case: Membership Validation
Context: The provider to validate a customer's membership to ChocAn.
Actors: Provider
Main Success Scenario:
1. When a card is read through a card reader on the terminal, the ChocAn Data Center is dialed.
2. The ChocAn Data Center computer verifies the member number.
2.1 If valid, the word "Validated" appears on the one line display.
3. The provider has an option to bill ChocAn for health services provided.

Extensions:
2a. The card number is not valid.
2a.1. The word "Invalid" is displayed on the one line display indicating the number is invalid.
2a.2. The use case continues at step 1.
2b. The member is suspended.
2b.1. The word "Suspended" is displayed on the one line display indicating the member owes fees and has been suspended.
2b.2. The use case continues at step 1.

Bill ChocAn

Use Case: Bill ChocAn
Context: The provider intends to bill ChocAn for providing a health service provided.
Actors: Provider
Main Success Scenario:
1. The provider selects "Bill ChocAn".
2. The provider keys in the date of service provided in MM-DD-YYYY format.
3. The provider uses the Provider Directory to look up six-digit service code corresponding to the service provided.
4. The provider enters in the six-digit code.
5. The software displays the name of the service corresponding to the code entered for verification.
6. The provider verifies that the correct code has been entered.
7. The provider has the option to add comments about the service provided.
8. The software looks up the fee associated with the service provided.
9. The provider has a form for entering member name and number, date of service, service code, fee to be paid, and current date and time.

Extensions:
2a. The date is entered in the wrong format.
2a.1. The system informs the provider.
2a.2. The use case continues at step 2.
4a. The provider has entered in a nonexistent code.
4a.1. The system prints out an error message.
4a.2. The use case continues at step 3.
6a. The wrong code has been entered for the service that was provided.
6a.1. The provider informs the system that the wrong code has been entered.
6a.2. The use case continues at step 3.

Provider Directory

Use Case: Provider Directory
Context: The provider requests the Provider Directory.
Actors: Provider
Main Success Scenario:
1. The provider selects that they want an alphabetically ordered list of the services provided through ChocAn and their corresponding service code.
2. The Product Directory list is sent to the provider as an email attachment.

Reports Use Cases

Generate Reports

Use Case: Generate Report
Context: A ChocAn manager requests a report of services provided that week.
Actors: ChocAn Manager, Member, Provider
Main Success Scenario:
1. A ChocAn manager selects that they want a report of services provided that week.
2. The manager select if they want a certain member's or provider's report, all the member's or provider's report, or all reports.
3. The type of report is generated by the ChocAn Data Center.
4. The report is sent to the manager.

Extensions:
3a. At midnight on Friday, the ChocAn Data Center reads the week's file of services provided and generates reports. The manager receives a summary report, and providers and members receive their respective reports as well.
3b. EFT data is generated and saved so it is accessible by 3rd party accounting software.

Member Management Cases

Member Management

Use Case: Create/Read/Update/Delete Members or Providers
Context: The ChocAn Data Center runs in interactive mode during the day for opperators to view or alter members or providers.
Actors: ChocAn Data Center operators
Main Success Scenario:
1. A ChocAn operator selects to add, delete, or update a member/provider.
2. The operator enters in the member/provider to be added, deleted, or updated.
3. The operator adds, deletes, or updates the member/provider.

Extensions:
2a. The member/provider entered by the operator to be deleted or updated is not in the system.

2a.1. The software alerts the operator of the error.
2a.2. The use case continues at step 1.
2b. The member/provider entered by the operator to be added is already in the system.
2b.1. The software alerts the operator.
2b.2. The software takes the operator to the member's/provider's profile.
2b.3. The use case continues at step 3.

Glossary

Member- The member is the individual belonging to ChocAn who recieves the services provided by the affiliated providers.

Member report- The member report is a generated report given weekly of the services recieved from the member sorted in chronological order by date.This report includes the member name, member number, member address, member city, state, zip code, and the service code.

Summary Report- The summary report is a weekly report given to the manager for accounts payable. The report lists every provider to be paid that week, the number of consultations each had, and his or her total fee for that week. Finally, the total number of providers who provided services, the total number of consultations, and the overall fee total are printed.

Manager- The manager handles and recieves both the summary report and the members report containing the information about the member and services.

Member Records- The member records contains the member's Name, service number, and service codes in order of date for which the member has recieved the service. These are used to keep track of billing information for both the member and the provider.

Acme Accounting Services- A third-party organization contracted out for processing payments of membership fees.

Operators- ChocAn employees that oversee member and provider records.

Provider(s)- Health Care professional who proveds services to ChocAn members.

Provider Directory- Is used to look up service codes corresponding to services provided.

Provider Records- Records of services provided by providers to customers.

Provider Report- A weekly report sent to providers who bill ChocAn that week containing the list of services he or she provided.
